## 1. 目的

ブログ「naoki-blog.com」・X（Twitter）・Substackメルマガの運営業務を、
**名前と役割を持った複数のAI社員（サブエージェント）が並列で働く「AI編集部」**として見える化する。

- 社長が「朝会」と合図すると、編集長がAI社員を並列稼働させ、成果をまとめて報告する
- 公開系の操作（ブログ公開・X投稿・メルマガ）は必ず社長の承認を挟む
- 将来的に定期自動実行（Phase 2）へ拡張できる構成にする

- 各AI社員は自分の名前と役職を自覚し、報告の冒頭に名乗る（例：「ミナトです。今週の記事案を3本提案します」）
- 性格付けは軽め（業務の邪魔をしない範囲）。あくまで業務品質が最優先

## 3. 仕組み（アーキテクチャ）

### 3-1. AI社員の定義ファイル

Claude Codeのカスタムエージェント機能を使い、`ai-company/.claude/agents/` に社員1名につき1ファイルを配置する。

```
ai-company/.claude/agents/
├── minato-research.md   （ミナト：リサーチ担当）
├── haru-writer.md       （ハル：執筆担当）
├── aoi-editor.md        （アオイ：校閲担当）
├── tsumugi-sns.md       （ツムギ：SNS担当）
├── riku-seo.md          （リク：SEO分析担当）
└── yume-newsletter.md   （ユメ：メルマガ担当）
```

各ファイルには以下を記載する：

- 名前・役職・一人称・報告時の名乗り方
- 担当業務の詳細な手順（どのファイル/データを見て、何をどこへ出力するか）
- やってはいけないこと（公開操作の禁止、体験談の捏造禁止 など）
- 成果物の出力先（`編集部/` フォルダ内のパス）

### 3-2. 起動コマンド「朝会」

`ai-company/.claude/commands/asakai.md`（スラッシュコマンド）を作成する。

- 社長が `/asakai` と入力（または「朝会」と発言）すると起動
- 編集長は次の順で動く：
  1. `編集部/タスクボード.md` と `記事フォルダ（記事/*.docx）` を確認
  2. **全AI社員をバックグラウンドで並列起動**（Phase 1では毎回全員）
  3. 各社員の完了を待ち、成果物を確認
  4. 社長へ「本日の朝会報告」として一覧で報告
  5. 承認が必要な項目（公開・投稿）を明示して社長の判断を仰ぐ
  6. `編集部/日報/YYYY-MM-DD.md` に活動記録を保存

### 3-3. 編集部の共有フォルダ

```
ai-company/編集部/
├── タスクボード.md    ← 今週のタスクと進捗。編集長が更新、社長も自由に編集可
├── 記事案リスト.md    ← ミナトの提案ストック。ステータス：提案中／採用／却下
├── 下書き/            ← ハル・ツムギ・ユメの成果物置き場
│   ├── ブログ/
│   ├── X投稿/
│   └── メルマガ/
└── 日報/              ← 朝会ごとの活動記録（YYYY-MM-DD.md）
```

- すべてMarkdownファイル。DBやサーバーは使わない（追加費用ゼロ）
- gitで履歴管理される（編集部の活動履歴が自動で残る）

## 4. 業務フロー

### 4-1. ブログ記事（docx持ち込みルート・現行踏襲）

1. 社長が `記事/` フォルダにdocxを置く
2. 朝会でハルがdocxを検出→Markdown下書きを `編集部/下書き/ブログ/` に作成
3. アオイが校閲（誤字・事実関係・YMYL観点・体験談挿入位置の提案）
4. 編集長が社長に提示 → **社長の承認後に** `src/content/blog/` へ配置し、ビルド確認、git push
5. 公開後、ツムギがX告知文を下書き → 社長承認後にX投稿ツールへ登録

### 4-2. ブログ記事（AI提案ルート・新設）

1. ミナトがトレンド・検索需要を調査し、記事案を `記事案リスト.md` に追加
2. 社長が記事案を見て「採用／却下」を決める
3. 採用案はハルが下書き→以降は4-1の3〜5と同じ
4. 体験談・実体験は**必ず社長へのヒアリングで取得**（AIは創作しない）

### 4-3. X投稿・メルマガ

- ツムギ：記事告知以外のX投稿ネタも下書きし `編集部/下書き/X投稿/` に保存。**投稿・予約登録は社長承認後のみ**
- ユメ：公開済みブログ記事をメルマガ向けに再編集し `編集部/下書き/メルマガ/` に保存。**Substackへの入稿・送信は社長が手動で行う**

### 4-4. SEO分析

- リクはSearch Consoleのエクスポートデータ（社長がダウンロードして共有）やサイトの状態を分析し、改善提案を朝会で報告
- 提案の実施（記事修正など）は採用されたものだけをハル／編集長が実行

## 5. 承認ゲート（絶対ルール）

AI社員・編集長が**承認なしで実行してよいこと**：

- リサーチ、下書き作成、校閲、分析、`編集部/` フォルダ内のファイル更新

**社長の承認が必須なこと**：

- ブログ公開（`src/content/blog/` への配置と git push）
- X投稿・投稿予約の登録
- メルマガの入稿・送信
- 記事案の採用決定
- 体験談・実体験の記事への追加（内容は社長ヒアリングでのみ取得）

**恒久的な禁止事項**（CLAUDE.mdの社内ルールに準拠）：

- APIキー・パスワード等をコードやMarkdownに直接書くこと
- 個人情報・未公開情報のリポジトリへのコミット
- 体験談・数値の捏造（YMYL領域のため特に厳守）

## 6. 段階的な展開計画

### Phase 1（今回実装する範囲）

- AI社員6名の定義ファイル作成
- `/asakai` コマンド作成（全員並列稼働）
- 編集部フォルダの初期構築
- **毎回全員を起動**し、どれだけ成果が出るか・トークン消費がどれくらいかを観察する

### Phase 2（Phase 1の運用結果を見て検討）

- 定期自動実行（スケジュール機能で毎朝自動的に朝会を開催）
- 承認ゲートは維持（自動公開はしない）
- トークン消費の観察結果次第で「必要な社員だけを起こす」省エネモードを設計

## 7. コストに関する申し送り

- AI社員の並列稼働はトークン消費が大きい（1回の朝会で社員1名あたり数千〜数万トークン）
- Phase 1では毎回の朝会報告に**概算のトークン消費**を記載し、社長がコスト感を掴めるようにする
- 消費が想定より大きい場合、Phase 2で省エネモード（必要な社員のみ起動）を優先検討する

## 8. エラー時の扱い

- ある社員の作業が失敗した場合、編集長は他の社員の成果はそのまま報告し、失敗した業務は「失敗した理由と再実行の提案」を添えて社長に報告する
- 同じ失敗を無断で何度もリトライしない（トークンの無駄遣い防止）

## 9. 成功の判断基準

- 「朝会」の一言で、記事下書き・X告知文・SEO報告などが一度に揃って報告される
- 社長の作業が「判断（承認・却下）」に集中できるようになる
- 承認なしに公開系の操作が一度も起きないこと（安全性の実証）
