# Notion「編集部ボード」設計書

- 日付: 2026-07-08
- ステータス: 社長承認済み

## 背景と課題

編集部の管理情報が以下の3か所に分散し、管理しづらくなっている。

- `編集部/タスクボード.md` … 進行中タスクの表。1行が長文化して読みにくい
- `編集部/記事案リスト.md` … 記事案とステータス。縦に長い
- `編集部/下書き/`（ブログ・X投稿・メルマガ） … ファイルはあるが「投稿済みか」「配信済みか」の状態はタスクボードや日報に散在

社長（非エンジニア）が挙げた課題は次の3点。

1. **承認待ちが分からない** … 社長が判断すべきものが一覧できない
2. **状態の記録がバラバラ** … 投稿済み・配信済み等の状態が長文の中にしか書かれていない
3. **編集の仕方が難しい** … Markdownの表や書式を崩さず編集するのが不安

## 決定事項

- 管理の場所は **Notion**（社長がブラウザ・スマホアプリで直接閲覧・編集）
- 下書きの**本文もNotionのページに転記**し、スマホだけで「読む→承認」が完結する
- 原稿のMarkdownファイルは従来通りリポジトリに残す（ブログ公開に必要）。**Notionは管理と承認の場**であり、カード本文は原稿のコピー

## 全体像

Notionに「編集部ボード」データベースを1つ作り、記事案・ブログ下書き・X投稿・メルマガ・SEOタスクをすべてカードとして一元管理する。

- 社長 → Notionでカードを読み、ステータス変更・メモ記入
- AI編集部 → 朝会のたびにNotion（claude.ai Notion連携＝MCP）を読んで社長の判断を拾い、成果物をカードとして追加・更新
- マスター（正）はNotion。タスクボード.md と記事案リスト.md は廃止

## データベース構造

データベース名: **編集部ボード**

| プロパティ | 型 | 内容 |
|---|---|---|
| タイトル | title | 例:「X投稿：パスキーの単発ネタ」「記事案：SIMスワップ詐欺」 |
| 種別 | select | 記事案 / ブログ下書き / X投稿 / メルマガ / SEO・サイト改善 / その他 |
| ステータス | select | 提案中 / 作業中 / 社長の確認待ち / 承認済み / 完了 / 却下 / 保留 |
| 担当 | select | ミナト / ハル / アオイ / ツムギ / リク / ユメ / 編集長 / 社長 |
| 日付 | date | 提案日・作成日 |
| 関連ファイル | rich_text | リポジトリ内の下書きファイルのパス |

ページ本文（ブロック）には以下を入れる。

- 下書きの本文（X投稿文・メルマガ原稿・記事案の概要など）
- 校閲メモ・補足（あれば）
- 「社長メモ」欄（社長が自由に追記する場所として見出しを用意）

### ステータスの意味（全種別で共通）

| ステータス | 意味 |
|---|---|
| 提案中 | ミナトの記事案など、採用判断前 |
| 作業中 | AI社員が執筆・調査中 |
| 社長の確認待ち | 社長の承認・判断・ヒアリング回答が必要 |
| 承認済み | 社長OK済み。公開・投稿・配信の実行待ち |
| 完了 | 公開・投稿・配信まで済んだ |
| 却下 / 保留 | 社長判断で見送り／一時停止 |

## ビュー

1. **📥 社長の承認待ち**（既定ビュー） … ステータス「提案中」または「社長の確認待ち」を表示（記事案の採用判断も社長の判断事項のため）。社長は普段これだけ見ればよい
2. **看板ボード** … ステータス別のカンバン表示
3. **種別ごと一覧** … 種別でグループ化した一覧

## 運用フロー

1. 社長は好きなタイミングでNotionを開き、承認待ちカードを読んでステータスを変更（承認済み／却下など）。意見は本文の「社長メモ」に追記
2. 朝会の冒頭で編集長がNotionを読み、社長の変更・メモを拾ってAI社員に指示（従来の「タスクボードの指示メモ確認」を置き換え）
3. 朝会の成果（記事案・下書き・X投稿案・メルマガ原稿）は編集長がカードとして追加・更新し、判断が必要なものは「社長の確認待ち」にする
4. ブログ公開・X投稿・メルマガ配信の実行は従来通り社長の明示的承認のもとClaude側で行い、済んだらカードを「完了」に変更

社長のNotion上の変更がAI編集部に反映されるのは「次に朝会またはチャットでClaudeが動いたとき」であり、常時同期ではない。

## 既存資産の移行と変更

| 対象 | 扱い |
|---|---|
| タスクボード.md の「進行中」タスク | Notionカードへ移行 |
| 記事案リスト.md の全記事案 | Notionカードへ移行（却下・執筆済み含む。ステータスは新体系に読み替え：提案中→提案中、採用→承認済み、執筆済み→完了、却下→却下） |
| X投稿・メルマガの未投稿/未配信分 | 状態を調べてカード化（投稿済み・配信済みは「完了」） |
| タスクボード.md／記事案リスト.md | 移行後は「Notionに移行しました」の案内とリンクだけ残す |
| `.claude/commands/asakai.md` | 手順1（状況確認）と手順5（更新）をNotion読み書きに変更 |
| `.claude/agents/*.md`（6名） | タスクボード・記事案リスト参照箇所をNotion参照に変更 |
| 下書きフォルダ・日報・編集部のGit自動保存 | 現状維持 |

## エラー時・例外時の扱い

- Notion接続（MCP）が使えない場合、朝会は従来のファイル情報のみで続行し、「Notionが読めなかった」ことを社長に報告する。勝手に別の管理ファイルを作らない
- 移行やカード更新に失敗したカードがあれば、朝会報告に失敗内容を明記する

## セキュリティ・情報の取り扱い

- Notionは社外サービスのため、**外に置きたくない情報は書かない**。ブログ原稿・X投稿・メルマガは公開前提の内容なので転記してよい
- 体験談ヒアリングの生メモなど個人的な内容はNotionに書かず、従来通りローカルファイル・チャットで扱う
- APIキー等の秘密情報は扱わない（Notion連携はclaude.aiの既存接続を使う）

## テスト・受け入れ確認

1. データベースと3ビューが作成され、社長のNotionから見える
2. 既存のタスク・記事案・未処理下書きがカードとして移行され、件数と内容が元ファイルと一致する
3. 社長がスマホでステータスを変更→次の朝会で編集長がその変更を拾って報告できる
4. 朝会後、新しい成果物がカードとして追加され「社長の確認待ち」ビューに並ぶ
5. タスクボード.md・記事案リスト.md が案内のみになり、朝会が新手順で完走する
